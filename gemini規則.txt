## 4. Auto-Versioning & Git Commit Protocol

- **Post-Edit Commit**: After completing any user-requested file modification in a git-initialized project, automatically:
  1. Stage all changed files: `git add -A`
  2. Generate a semantic commit message in Conventional Commits format (e.g., `feat: add user dashboard component`, `fix: resolve color detection edge case`)
  3. Commit locally: `git commit -m "<message>"`
  4. **ASK the user** before pushing: Display the commit summary and ask "是否要推送至遠端？" before executing `git push`
  
- **Version Bump**: After each functional change:
  1. Read the current version from `package.json`, `pyproject.toml`, or `PROJECT_STATUS.md`
  2. Bump the PATCH version (e.g., `1.0.2` → `1.0.3`) for fixes, MINOR for features, MAJOR for breaking changes
  3. Update the version field in the source file and `PROJECT_STATUS.md`
  
- **Changelog Entry**: Append a new row to the "版本歷程" table in `PROJECT_STATUS.md` with:
  - Version number | Date (YYYY-MM-DD) | Change type (feat/fix/refactor/docs) | Description in Traditional Chinese
